## Problem Statement

XORing의 `/admin`은 현재 정적 호스팅 전제에서 동작하는 읽기 전용 관리자 셸이다. 실제 운영 목적은 `frontier_applications`에 쌓인 신청 데이터를 확인하는 것이지만, 현재 UI는 데이터 길이에 비해 여백이 크고 정보 밀도가 낮아서 운영자가 빠르게 훑기 어렵다. 이번 재설계의 목표는 기능 추가보다 `가독성`, `사용성`, 그리고 앞으로의 관리자 디자인 시스템 기준점을 만드는 것이다.

## Demand Evidence

- 운영자는 현재 `/admin`에서 목록만 본다. 즉, 이 화면의 실제 가치는 "처리"가 아니라 "빠른 스캔"에 있다.
- 사용자는 현재 UI에 대해 "데이터가 눈에 잘 안 들어와", "데이터 길이 대비 여백이 많아서 공간을 효율적으로 쓰지 못한다"고 명확히 말했다.
- 가장 중요한 읽기 우선순위도 구체적으로 정해졌다: `최근 신청일 + 이메일 + 상태`.
- 실제 스캔 순서에 대한 관찰도 나왔다: 운영자는 처음 3초에 `이메일`과 `최근 신청일`을 먼저 찾는다.

## Status Quo

- 현재 구조는 정적 `/admin` 라우트 + Supabase Auth + `admin_users` allowlist + RLS 직접 조회다.
- 로그인 후에는 요약 카드와 표가 섞인 최소 셸이 보이지만, 표 밀도가 낮고 핵심 정보 우선순위가 약하다.
- 모바일까지 고려하면 현재 테이블 중심 레이아웃은 더 쉽게 무너질 가능성이 크다.
- 즉, 기술 경계는 이미 맞는데, 정보 설계와 시각 시스템이 운영 목적에 비해 아직 거칠다.

## Target User & Narrowest Wedge

### Target user

- 내부 운영자 또는 검토 담당자
- 해야 하는 일: 신청 목록을 빠르게 훑고 현재 상태를 파악한다
- 가장 먼저 보는 정보: 이메일, 최근 신청일, 상태

### Narrowest wedge

- `/admin`을 "신청자 인덱스"로 재정의한다.
- 1차 범위는 `사이드바 + 상단 요약 + 검색/상태 필터 셸 + 고밀도 목록 + 모바일 카드 전환`이다.
- 차트, 상세 편집, 승인 처리, 복잡한 액션은 이번 범위에 넣지 않는다.

## Constraints

- 정적 호스팅 유지. `/admin`도 같은 Next.js 정적 앱 안에서 동작해야 한다.
- Supabase Auth, `admin_users` allowlist, `frontier_applications` 조회 구조는 유지한다.
- 기존 디자인은 무시해도 되지만, 현재 데이터 로딩/권한 경계는 깨지면 안 된다.
- shadcn 템플릿 기반 구성이어야 하며, sidebar는 필수다.
- 모바일도 지원해야 하지만, 데스크톱 읽기 성능을 희생하는 방식이면 안 된다.

## Premises

1. 이 화면은 분석 대시보드가 아니라 `신청자 인덱스`다.
2. 이번 1차 목표는 기능 확장보다 읽기 성능이 좋은 기준 화면을 만드는 것이다.
3. 모바일 대응은 데스크톱 테이블 축소가 아니라 핵심 3요소 중심의 별도 카드 레이아웃으로 가야 한다.

## Approaches Considered

### Approach A: Dense Index

데스크톱에서 고밀도 테이블을 중심에 두고, 좌측 사이드바는 최소 탐색 역할만 수행하는 방식이다. 가장 직접적으로 여백 문제를 해결하고 읽기 속도를 높일 수 있다. 다만 기준 화면으로서의 확장 슬롯은 약하고, 이후 필터/상세 패턴을 따로 덧붙여야 한다.

### Approach B: Filtered Workbench

사이드바, 상단 요약, 검색/상태 필터 바, 주 리스트를 균형 있게 배치하는 운영형 셸이다. 리스트는 충분히 촘촘하게 유지하되, 이후 상세 뷰나 추가 메뉴로 확장할 수 있는 골격을 같이 만든다. 지금 합의된 `A + C` 요구를 가장 잘 만족한다.

### Approach C: Mobile-First Stack

모바일 카드 레이아웃을 먼저 정의하고 데스크톱에서 확장하는 방식이다. 모바일 적응력은 가장 좋지만, 현재 실제 사용 장면이 데스크톱 스캔 중심이므로 우선순위가 과할 수 있다.

## Recommended Approach

`Filtered Workbench`를 채택한다.

이 접근은 세 가지를 동시에 만족한다.

- 지금 당장 필요한 `읽기 성능`
- shadcn 기반의 `사이드바형 관리자 기준 화면`
- 이후 다른 admin 화면으로 퍼질 수 있는 `확장 가능한 디자인 시스템 뼈대`

핵심 설계 방향은 다음과 같다.

- 왼쪽 sidebar는 "현재 위치와 앞으로 붙을 메뉴"를 보여주는 얇은 내비게이션 셸로 유지한다.
- 본문 상단은 감성형 히어로가 아니라 작은 KPI 요약과 빠른 필터 영역으로 구성한다.
- 실제 무게 중심은 고밀도 신청 목록이다. 데스크톱에서는 한 화면에 더 많은 행이 보여야 하고, 모바일에서는 같은 정보를 카드형으로 압축해 수평 스크롤 없이 읽혀야 한다.
- `최근 신청일`, `이메일`, `상태`를 시선이 가장 먼저 닿는 위치에 둔다. 나머지 컬럼은 보조 정보로 낮춘다.

## Open Questions

- 1차 필터 셸에 실제 동작까지 포함할지, 아니면 UI 슬롯만 둘지는 구현 단계에서 결정이 필요하다.
- 모바일에서 `wallet_address`, `payment_token` 같은 보조 필드는 기본 접힘으로 둘지, 두 줄 압축으로 보일지 선택이 필요하다.
- 로그인 화면도 같은 admin 디자인 시스템 톤으로 올릴지, 현재처럼 별도 단일 패널로 둘지는 후속 결정 사항이다.

## Success Criteria

- 데스크톱 `/admin`에서 현재보다 더 많은 신청 행이 첫 화면에 보여야 한다.
- 운영자는 첫 3초 안에 `최근 신청일`, `이메일`, `상태`를 무리 없이 훑을 수 있어야 한다.
- 상단 요약과 필터 셸은 목록을 방해하지 않고 보조해야 한다. 즉, 시선이 표에 도달하기 전에 피로를 만들면 실패다.
- 모바일에서는 수평 스크롤 없는 카드/압축 레이아웃으로 동일 핵심 정보를 읽을 수 있어야 한다.
- 새 레이아웃은 이후 admin 상세/설정/다른 목록 화면으로 확장 가능한 톤과 간격 규칙을 제공해야 한다.

## Distribution Plan

이 화면은 기존 정적 배포 파이프라인을 그대로 사용한다.

- `next.config.ts`의 static export를 유지한다.
- `npm run build` 결과물 `out/`에 `/admin`과 `/admin/login`이 포함된다.
- 별도 백엔드 배포나 새로운 배포 채널은 필요 없다.
- 기존 Supabase Auth + RLS 구조가 데이터 보호 경계를 계속 담당한다.

## Dependencies

- 기존 `src/components/admin/AdminDashboardPanel.tsx`의 데이터 로드 구조 재사용
- shadcn 기본 컴포넌트와 sidebar 패턴 도입 또는 추가 설치
- 현재 `status` 배지/카운트 로직 재구성
- 모바일 대응용 목록 표현 패턴 결정

## The Assignment

실제 운영자 1명과 함께 현재 `/admin`을 5분만 보면서, 아무 설명 없이 "처음 10초에 어디부터 보는지"를 말하게 하라. 그때 시선이 멈추는 지점과 읽기 순서를 메모하라. 이번 설계는 그 실제 스캔 순서를 맞추는 데 써야지, 디자이너의 추측에 쓰면 안 된다.

## What I noticed about how you think

- 당신은 기능을 늘리자는 쪽으로 안 갔다. "admin 목록만 볼거고"라고 먼저 범위를 잘랐다.
- 미감보다 목적을 더 정확히 봤다. "디자인 시스템을 발전시키는게 주 목적이야"라고 해서 이번 작업을 기준 화면 설계로 재정의했다.
- 추상적으로 말하지 않고 관찰 가능한 문제를 짚었다. "데이터가 눈에 잘 안들어와", "여백이 많아서 공간을 효율적으로 쓰지 못하고"라고 했다.
- 정보 우선순위도 바로 정했다. "최근 신청일 + 이메일 + 상태"와 "이메일, 최근 신청일을 먼저 찾는다"는 말이 이번 설계의 핵심 기준이 됐다.
